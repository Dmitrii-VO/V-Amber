# Техническое задание: Amberry Voice (AI Live-Commerce Assistant)

## 1. Общее описание проекта
**Amberry Voice** — это интеллектуальная система автоматизации продаж для прямых эфиров (Live Commerce) в социальной сети ВКонтакте. Система позволяет ведущему управлять процессом продаж голосом, автоматически информировать зрителей о товарах и резервировать позиции в CRM-системе «МойСклад».

## 2. Цели и задачи
* **Освобождение ведущего:** Минимизация ручных действий во время эфира.
* **Мгновенная реакция:** Сокращение времени между объявлением товара и появлением информации в чате.
* **Автоматизация учета:** Исключение ошибок при резервировании и сборе данных клиентов.
* **Масштабируемость:** Возможность работы системы на различных платформах (Windows, macOS).

## 2.1. Границы MVP
В первый рабочий релиз входят:
* Голосовое объявление товара -> поиск товара -> публикация карточки в VK.
* Обработка комментария «Бронь» -> создание документа «Заказ покупателя» в «МойСклад».
* Telegram-управление: подтверждение неоднозначного кода товара, закрытие активного лота, отмена последней брони, критические уведомления.

Не входят в MVP:
* Автоматический сбор контактов новых клиентов.
* Работа с товарами, имеющими модификации (размер, цвет, SKU).
* Расширенный операторский Web UI с полной аналитикой и ручным управлением заказами.

## 3. Архитектура системы

### 3.1. Захват и распознавание речи (STT)
* **Источник звука:** Встроенный или внешний микрофон ноутбука (локальный захват).
* **Технология:** **Yandex SpeechKit (Streaming API)**.
* **Целевая платформа MVP:** Windows.
* **Логика:** Минимальный Web UI в браузере получает доступ к микрофону, захватывает аудио и передает поток в backend для отправки в Yandex SpeechKit в реальном времени.

### 3.2. Интеллектуальный парсинг (NLP/LLM)
* **Модель:** **YandexGPT 5 Lite**.
* **Функции:**
    * Извлечение числовых артикулов из спонтанной речи ведущего (например, «код товара триста три»).
    * Нормализация словесных числительных в числовой артикул.
    * Использование LLM только в случае, если regex/простые правила не дали уверенного результата.

**Опциональное усиление:** при недостаточном качестве fallback-модели допускается замена на **YandexGPT 5 Pro** без изменения общей архитектуры.

**Правило MVP:** для комментариев VK не используется LLM-классификация. Все комментарии логируются, действие запускается только по ключевому слову «Бронь».

### 3.3. Интеграции (API)
* **VK API (LongPoll):** Мониторинг комментариев в реальном времени и публикация ответов.
* **МойСклад API:** Проверка остатков по модификациям, получение цен, создание документа «Заказ покупателя».
* **Telegram Bot API:** Центр уведомлений и пульт управления для ведущего.

**Канал MVP во VK:** комментарии/чат трансляции VK Live.

## 4. Функциональные требования

### 4.1. Управление лотами
1.  Базовая фраза-маркер MVP: «Код товара [число]».
2.  Список голосовых триггеров должен быть настраиваемым в конфигурации, без hardcode в бизнес-логике.
3.  При обнаружении в речи фразы-маркера система извлекает числовой артикул и делает запрос в «МойСклад».
4.  Если извлечение артикула неуверенное или найдено несколько кандидатов, система отправляет в Telegram топ-3 кандидата и исходный фрагмент распознанной фразы для ручного подтверждения.
5.  В чат VK автоматически отправляется карточка товара: Название, Цена, Доступный остаток с учетом резервов, Фото. Если фото отсутствует, используется заглушка с логотипом Amberry.
6.  Цена лота фиксируется на момент объявления товара и используется далее при бронировании, даже если цена в «МойСклад» изменится во время эфира.
7.  Одновременно активен только один лот.
8.  Новый распознанный код автоматически закрывает предыдущий активный лот и открывает новый.
9.  Повторное произнесение того же кода приводит к повторной публикации карточки товара в VK.
10. Лот остается активным до появления нового кода или до ручной команды закрытия через Telegram.
11. Для каждого объявления товара создается уникальный `lotSessionId`. Он используется для аудита, связи с комментариями, бронями и waitlist.
12. Команда закрытия лота через Telegram:
    * снимает активность лота;
    * блокирует прием новых броней по нему;
    * публикует сообщение в VK: «Лот закрыт».

### 4.2. Обработка бронирований (Бронь)
1.  В MVP система отслеживает только точное ключевое слово «Бронь».
2.  Бронь принимается только пока лот активен.
3.  Если один и тот же пользователь повторно пишет «Бронь» по одному и тому же активному лоту, дубликаты молча игнорируются.
4.  Если несколько зрителей отправили «Бронь» почти одновременно, приоритет определяется по времени комментария VK: first-comment-wins.
5.  В MVP поддерживаются только простые товары без модификаций. Товары с размерами, цветами и SKU выносятся за пределы MVP.
6.  При наличии товара:
    * В «МойСклад» сразу создается документ «Заказ покупателя» с резервом на 1 единицу.
    * В заказе должны быть заполнены поля: Организация, Склад, Статус, Канал продаж, Комментарий.
    * Склад выбирается из настройки программы для текущего эфира.
    * Покупатель получает ответ в VK: «@name, товар забронирован! Инструкции в ЛС.»
7.  При отсутствии товара:
    * Пользователь ставится в «тихий» лист ожидания без публичного ответа в VK.
    * Запись waitlist сохраняется в Redis.
    * В Telegram отправляется уведомление оператору.
    * Очередь ожидания хранится по `product id` из «МойСклад».
8.  После отмены ранее созданной брони оператор в Telegram должен иметь возможность вручную нажать кнопку «Предложить следующему» для первого участника очереди ожидания.

### 4.3. Работа с новыми контрагентами
* В MVP поиск контрагента в «МойСклад» выполняется сначала по `VK ID`, затем по имени/нику из VK.
* `VK ID` должен храниться в пользовательском поле (Custom Field) контрагента.
* Если контрагент не найден, он автоматически создается на основе данных VK.
* Обязательные поля нового контрагента в MVP: `name`, `VK ID`, `source=VK`.
* Шаблон имени нового контрагента: `VK: <First Name> <Last Name>`.
* Автоматический сбор контактов (телефон, адрес доставки) выносится за пределы MVP.

## 5. Технический стек
* **Языки:** TypeScript (Node.js) для основной логики, Python для работы со звуковыми драйверами (при необходимости).
* **Интерфейс MVP:** Минимальный локальный Web UI в браузере + Telegram Bot API.
* **Runtime-state:** Redis для активного лота, очередей, последней брони, временного состояния сессии.
* **Аудит и история:** SQLite.
* **Развертывание MVP:** Локально через Docker Compose.
* **Конфигурация и секреты:** `.env` и Docker environment variables.

## 5.1. Минимальный Web UI MVP
Web UI в MVP нужен только для задач, связанных с локальным микрофоном и оперативным контролем сессии.

Обязательные функции:
* Выбор устройства микрофона.
* Старт и остановка аудиостриминга.
* Отображение realtime-транскрипта.
* Отображение текущего активного лота.
* Подтверждение неоднозначного артикула, если Telegram недоступен или оператор работает с ноутбука.

Необязательные функции MVP:
* Расширенная аналитика.
* Управление waitlist.
* Ручное редактирование заказов.

## 5.2. Хранение данных и аудит
Обязательно сохраняются:
* Распознанные команды и артикулы.
* Комментарии VK.
* Действия по броням и отменам.
* Ошибки внешних API.

Дополнительно:
* Сырые транскрипты речи не входят в MVP.
* Технические метрики и задержки собираются в минимальном объеме.
* Retention для истории и аудита: 90 дней.

Минимальный набор сущностей SQLite:
* `lot_sessions`
* `vk_comments`
* `reservations`
* `waitlist_entries`
* `api_errors`
* `telegram_actions`

## 5.3. Отказоустойчивость
* **VK API / МойСклад / YandexGPT:** до 3 повторных попыток, затем логирование и уведомление в Telegram.
* **Yandex SpeechKit:** без retry в MVP, логирование и уведомление в Telegram.

## 5.4. Наблюдаемость
* Консольный вывод при локальном запуске.
* Telegram-уведомления для критических ошибок и ручных действий оператора.
* Минимальный Web UI для локального контроля микрофона и статуса сессии.

## 6. Сценарии использования

### Сценарий А: Стандартная продажа
1.  Ведущий: «Перед вами янтарный кулон, код товара 402».
2.  Система: Находит товар, фиксирует `lotSessionId`, публикует данные в чат ВК.
3.  Зритель: «Бронь!».
4.  Система: Ищет или создает контрагента, создает «Заказ покупателя» в «МойСклад» и подтверждает бронь в чате.

### Сценарий Б: Управление через Telegram
1.  Ведущий замечает ошибку в коде или хочет отменить последнюю бронь.
2.  В Telegram-боте доступны подтверждение неоднозначного артикула, закрытие активного лота и отмена последней брони.
3.  При отмене брони система использует последнюю созданную бронь из runtime-state, выполняет допустимую в «МойСклад» операцию отмены/обновления статуса и уведомляет об этом.
4.  Если по товару есть waitlist, оператор получает возможность вручную предложить товар следующему участнику очереди.

## 7. План разработки (Roadmap)
1.  **Этап 1:** Настройка стриминга аудио в Yandex SpeechKit.
    * Выбор микрофона.
    * Старт/стоп стриминга.
    * Потоковая отправка аудио в Yandex SpeechKit.
    * Вывод realtime-транскрипта.
    * Логирование ошибок.
    * Минимальные метрики задержки.
2.  **Этап 2:** Извлечение артикулов из речи.
    * Regex/простые правила для фразы «Код товара [число]».
    * Нормализация словесных числительных.
    * Подключение YandexGPT 5 Lite как fallback.
    * Telegram-подтверждение неоднозначных кандидатов.
3.  **Этап 3:** Интеграция с «МойСклад».
    * Поиск товара.
    * Получение цены, фото и доступного остатка.
    * Фиксация цены на момент объявления.
    * Поиск/создание контрагента по `VK ID` и имени.
    * Создание документа «Заказ покупателя».
4.  **Этап 4:** Интеграция с VK Live.
    * Публикация карточки товара.
    * Отслеживание комментариев «Бронь».
    * Подтверждение брони в чате.
    * Работа с активным лотом и `lotSessionId`.
5.  **Этап 5:** Telegram-операции и аудит.
    * Закрытие активного лота.
    * Отмена последней брони.
    * Критические уведомления.
    * Waitlist и ручное предложение следующему.
    * Аудит в SQLite.
6.  **Этап 6:** Минимальный Web UI.
    * Выбор микрофона в браузере.
    * Старт/стоп локального аудиостриминга.
    * Просмотр realtime-транскрипта.
    * Отображение активного лота.
    * Подтверждение неоднозначного артикула.

## 8. Критерии готовности

### 8.1. Готовность этапа 1
Этап 1 считается завершенным, если система:
* позволяет выбрать микрофон;
* умеет запускать и останавливать стриминг;
* отправляет поток аудио в Yandex SpeechKit;
* показывает realtime-транскрипт;
* логирует ошибки;
* собирает минимальные метрики задержки.

Статус на текущий момент: **критерии Этапа 1 подтверждены локальной
проверкой**.

Подтверждено в рабочем локальном контуре:
* выбор микрофона в браузере работает;
* старт и остановка стриминга работают;
* поток передается в Yandex SpeechKit Streaming API;
* realtime-транскрипт возвращается в Web UI;
* ошибки пишутся в `logs/server.log` и отображаются в UI;
* минимальные метрики задержки выводятся в интерфейсе.

Примечание по текущей рабочей конфигурации SpeechKit:
* локальная проверка успешно подтверждена в режиме без принудительной передачи
  `x-folder-id`;
* если используется `YANDEX_SPEECHKIT_SEND_FOLDER_HEADER=1`, значение
  `YANDEX_SPEECHKIT_FOLDER_ID` должно точно соответствовать scope
  используемого `YANDEX_SPEECHKIT_API_KEY`.

### 8.2. Готовность MVP
MVP считается готовым, если выполняется следующий сценарий:
1.  Ведущий произносит: «Код товара 402».
2.  Система находит товар и публикует карточку в VK Live.
3.  Зритель пишет комментарий: «Бронь».
4.  Система создает «Заказ покупателя» в «МойСклад».
5.  Оператор может отменить последнюю бронь через Telegram.
6.  Все ключевые события сохраняются в аудит.

## 9. Текущий статус реализации
На текущем этапе реализован рабочий локальный контур для проверки **Этапа 1**:
browser Web UI -> Node.js backend -> Yandex SpeechKit Streaming API ->
realtime-транскрипт обратно в Web UI.

### 9.1. Что уже реализовано
Подтверждено локальной проверкой:
* Выбор устройства микрофона в браузере.
* Старт и остановка локального аудиостриминга.
* Потоковая передача PCM `16 kHz`, `LINEAR16_PCM`, `1 channel` в backend.
* Проксирование потока в **Yandex SpeechKit (Streaming API)**.
* Отображение `partial` и `final` транскриптов в Web UI.
* Базовые метрики задержки для `partial` и `final` событий.
* Логирование server-side событий и ошибок в файл `logs/server.log`.

### 9.2. Текущий локальный runtime
Для локальной проверки используется минимальный runtime без Docker Compose:
* Web UI: статические файлы в каталоге `web-ui/`.
* Backend: Node.js server в каталоге `server/`.
* HTTP endpoint: `http://localhost:8080/`.
* Health endpoint: `http://localhost:8080/health`.
* WebSocket endpoint для STT: `ws://localhost:8080/ws/stt`.

Запуск локального контура:
1.  Выполнить `npm install`.
2.  Выполнить `npm start`.
3.  Открыть `http://localhost:8080/` в Chrome или Edge.

### 9.3. Формат логов
Файл `logs/server.log` ведется в формате JSONL: одна JSON-запись на строку.
Такой формат удобен для последующего анализа человеком, grep-поиска и
автоматической обработки ИИ.

Минимальные типы записей:
* `server_started`
* `client_connected`
* `stream_start_requested`
* `stream_opened`
* `final_transcript`
* `stream_stop_requested`
* `stream_closed`
* `stream_error`

### 9.4. Особенность конфигурации Yandex SpeechKit
Для текущего `YANDEX_SPEECHKIT_API_KEY` подтверждено, что принудительная
передача `x-folder-id` может приводить к ошибке несоответствия folder scope,
если `YANDEX_SPEECHKIT_FOLDER_ID` не совпадает с folder, к которому привязан
ключ.

Практическое правило текущей реализации:
* `YANDEX_SPEECHKIT_SEND_FOLDER_HEADER=0` или отсутствие переменной - безопасный
  режим по умолчанию.
* `YANDEX_SPEECHKIT_SEND_FOLDER_HEADER=1` допускается только после проверки,
  что `YANDEX_SPEECHKIT_FOLDER_ID` точно соответствует scope используемого
  `YANDEX_SPEECHKIT_API_KEY`.
