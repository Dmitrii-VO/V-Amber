## Логические ошибки

### Исправлено

Ниже отмечены логические ошибки, которые уже исправлены в коде.

- [x] **`ws-server.js`** — устаревшее подтверждение из `Telegram`
  больше не может открыть старый лот после смены run или нового
  actionable detection.
- [x] **`ws-server.js`** — исправлена гонка между `activeLot` и
  `currentLot` в VK comment polling: бронь теперь привязывается к тому
  же лоту, для которого читались комментарии.
- [x] **`ws-server.js:218-226`** — комментарии `"бронь"`, написанные
  сразу после публикации карточки лота, больше не теряются при
  инициализации polling.
- [x] **`ws-server.js`** — после сбоя `MoySklad` пользователь снова
  может повторить бронь: `viewerId` удаляется из `acceptedUserIds`.
- [x] **`ws-server.js`** — waitlist больше не зависает после падения
  первичной брони: следующий ожидающий event автоматически
  переобрабатывается, пока лот ещё активен.

### Критические

- [x] **`article-extractor.js:158-184`** — выход за границы массива в `parseSubThousand()`: доступ к `words[consumed]` без проверки `consumed < words.length`. Добавить явную проверку перед каждым обращением к элементу.
- [x] **`article-extractor.js:298`** — выход за границы в цикле парсинга чисел: после `cursor += parsed.consumed` возможен `remainingWords[cursor]` при `cursor >= length`. Добавить проверку `cursor < remainingWords.length` перед `isNumericWord(...)`.
- [x] **`ws-server.js:132-187`** — `state.primaryReservation` не сбрасывается после успешного создания заказа (только в `catch`). После первой брони блокировка остаётся навсегда — все последующие "бронь" получают `waitlist_pending` и не обрабатываются. При ошибке — пользователь не может повторить бронь, так как его ID остаётся в `acceptedUserIds`. Исправить: перенести `state.primaryReservation = null` в `finally`; при ошибке удалять viewerId из `acceptedUserIds`.

### Высокие

- [x] **`ws-server.js:439-530`** — состояние гонки при переключении лотов: если `runId` меняется в середине `handleConfirmedDetection()`, `activeLot` остаётся частично обновлённым (VK-публикация случилась, но не записана; опрос комментариев не стартует).
- [x] **`ws-server.js:550`** — `customerOrdersByViewerId = new Map()` сбрасывается при каждом нажатии "Старт". Если оператор перезапускает стрим в середине сессии — связи "зритель → заказ" теряются, и последующие брони создают новые заказы вместо добавления позиций к существующим.
- [x] **`ws-server.js:218-226`** — при инициализации опроса есть пауза 2 секунды, за которую фиксируется последний ID комментария. Комментарии "бронь", написанные в это окно сразу после появления карточки лота, будут пропущены.

### Средние

- [x] **`config.js:45-48`** — не проверяется что `minLength <= maxLength`; при инвертированных значениях `filterCandidatesByLength()` отфильтрует все кандидаты без ошибки.
- [x] **`config.js:51-55`** — отсутствие ключей YandexGPT молча отключает LLM-fallback (нет предупреждения при старте).
- [x] **`ws-server.js:194`** — хрупкое обнаружение фатальных ошибок VK API: поиск подстрок `"VK API 15"` и `"video not found"` не покрывает другие фатальные коды.
- [x] **`telegram.js:3-5`** — генерация токенов подтверждения через `Math.random()` — не криптографически безопасна; заменить на `crypto.getRandomValues()`.
- [x] **`ws-server.js:569-607`** — `runId` проверяется только в конце цикла детекции; при смене `activeRunId` в середине цикла `lastDetection` может быть записан для старой сессии.
