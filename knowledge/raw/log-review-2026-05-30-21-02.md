## Problems found

- VK publish failures at stream end. In the fresh session, comment polling
  stopped with `VK API 15: Access denied: video not found`; then each
  `lot_closed` publish failed for open lots. End-of-stream close should handle
  an ended or inaccessible video without flooding fatal publish errors.
- VK lot-card publish can fail with `VK API 100: ... photo is undefined`
  for product `00136`. The card publisher should not send an undefined photo
  parameter when the product has no usable image.
- Safe mode started on and blocked the first lot-card publish before the
  operator disabled it. The UI should make this state obvious before a stream
  starts, because it silently prevents external writes.
- Some MoySklad cards return unknown stock. Fresh session examples:
  `00269` and `00192` logged `stock_unknown_first_reservation`; the system
  accepted the first reservation despite unknown stock. This is a known
  oversell risk.
- Product code `00011` was detected by voice but not found in the product
  catalog. The operator needs a clear UI state for "code not found" during a
  live stream.
- Variant/modification products caused operator confusion: he observed that
  a modification showed under another code / parent product. Product lookup
  needs a variant-aware check so manual and voice code entry resolve the exact
  sellable item.
- Old `0.1.26` sessions show MoySklad timeout incidents:
  product lookup and counterparty lookup timed out after 8000 ms, causing at
  least one `reservation_order_failed` and one orphan waitlist at lot close.
  The fresh `0.1.33` session did not repeat these errors.
- Background product-code cache refresh failed repeatedly overnight with
  `fetch failed`. This is noisy and should be surfaced as degraded catalog
  refresh rather than repeated generic errors.
- Starting another server while port `8080` was already used produced
  `EADDRINUSE`. The launcher/runbook should keep the "port busy" recovery path
  easy for non-technical operators.

## Operator wishes and observations

- Manual code entry is a strong positive workflow. The operator repeatedly
  noted that typed codes appear almost instantly, while voice requires waiting
  for transcription. This suggests manual entry should be treated as a primary
  live-assistant workflow, not only a fallback.
- Voice detection latency can lose sales during active speech. The operator
  described that he must keep talking during a live stream, but the lot appears
  only after the transcript finalizes; in high demand this delay can be
  harmful.
- Cancellation still needs a faster operational path. Dashboard cancellation
  worked, but the operator does not want to scroll through many lots to find
  an old buyer/code. Requested shape: search/jump to reservation, voice cancel
  by buyer + code, and visible cancellation confirmation.
- Buyer-facing service comments should come from the official Amberry group,
  not the older Amber Standard/personal-looking identity. The operator wants
  confirmation messages to look official.
- Buyers should receive clear confirmation of which lot was reserved; buyers
  who did not get the item should be notified quietly, preferably by DM rather
  than more public comment noise.
- Wishlist/list behavior remains important. Earlier session text says "виш
  лист не работает походу" and asks to explain `СПИСОК <код>` / send the
  result privately or hidden, because public comment streams become hard to
  read.
- Multi-quantity reservation is still requested from the live workflow:
  examples include "две штуки" and "забронируй сразу две штуки". Existing
  buyer-comment quantity support does not fully cover the operator's live
  voice/manual-control expectation.
- The buyer command is still cognitively hard for some viewers. A viewer wrote
  "сложно" after the operator explained `бронь + код товара`; the operator
  wants the flow simplified where possible. Existing bare-code reservation
  helps, but older logs show `бронь 0588` did not match `00588`, so tolerant
  leading-zero normalization remains valuable.
- Price parsing and discount parsing need more hardening. Logs include
  repeated observations about `стоимость`, missing final digits, and an
  incorrect `10` price/discount interaction while testing percent discounts.
